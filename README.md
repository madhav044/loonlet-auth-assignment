# Loonlet Auth Assignment

A FastAPI-based authentication system with JWT Authentication and Role-Based Access Control (RBAC) developed as part of the Loonlet Innovations Pvt. Ltd. coding assessment.

---

## Features

### Authentication
- **User Registration API** — Secure account creation with automated role assignment.
- **User Login API** — Validates credentials and returns a secure session token.
- **JWT Token Authentication** — Stateless, secure session management.
- **Password Hashing** — Secure password storage utilizing strong one-way hashing algorithms.
- **Secure Authentication Flow** — Built following industry-standard OAuth2 security protocols.

### Role-Based Access Control (RBAC)

Supported System Roles:
- **Admin**
- **Therapist**
- **Parent**

### Protected Endpoints
- **Admin Portal** — Accessible exclusively by Admin users to view all registered users.
- **Therapist Portal** — Accessible exclusively by Therapist users for clinical endpoints.
- **Parent Portal** — Accessible exclusively by Parent users to view individual profile information.

## Core Architecture & Theoretical Concepts

### 1. RESTful API Architecture
- **Statelessness:** The API does not persist client session states on the server. Every incoming request must contain all credentials necessary to authenticate itself, ensuring the backend can scale horizontally.
- **Separation of Concerns:** Data persistence, business logic validations, and access controls are fully contained within the backend, independent of any downstream user interface.

### 2. Cryptographic Password Hashing
- **One-Way Cryptographic Salting & Hashing:** Plain-text passwords are never saved. Using Passlib with bcrypt, passwords undergo a secure, computationally expensive hashing process combined with a unique salt to mitigate rainbow table and brute-force attacks.
- **Verification Flow:** During a login event, the incoming plain-text password is compared against the stored hash safely without reversing the original string.

### 3. JWT (JSON Web Token) Security Structure

A valid JWT consists of three distinct, dot-separated blocks:

- **Header:** Explicitly defines the hashing algorithm (e.g., HS256) and the token type.
- **Payload:** Holds the public claims — the user's email ID, token expiration timestamp `exp`, and assigned role. *(Note: Payload data is base64-encoded, not encrypted. Sensitive items like raw passwords are never injected here.)*
- **Signature:** Generated by cryptographically mixing the header, payload, and a secure server-side Secret Key. This guarantees token integrity — if a client attempts to forge or alter their role, the server-side signature validation fails instantly.

### 4. Role-Based Access Control (RBAC)

The system enforces the **Principle of Least Privilege (PoLP)**. Permissions are mapped exclusively to dedicated system roles:

- **Admin Role:** Full global visibility. Can audit and view all registered system users.
- **Therapist Role:** Locked down to specialized endpoints managing therapist-specific workflows.
- **Parent Role:** Restricted tightly to personal profile metadata and dependent-level data views.

### 5. Dependency Injection Architecture

FastAPI's native `Depends` system handles access guardrails globally:

- **Token Extraction & Parsing:** Automatically intercepts the `Authorization: Bearer` token string from inbound HTTP request headers.
- **Inline Authorization Guards:** Before execution passes to the endpoint's core logic, sub-dependencies decode the token, check expiration, pull user data, and verify the user's role meets the target route's permissions. Any violation instantly triggers an automated `401 Unauthorized` or `403 Forbidden` response.

### 6. Relational Object Mapping (ORM)
- **SQLAlchemy Abstraction Layer:** Maps Python classes directly to database schemas, providing type safety, data integrity, and automatic SQL injection defence via built-in query parameterization.

## Installation & Setup

### Step 1 — Clone the Repository

```bash
git clone https://github.com/madhav044/loonlet-auth-assignment.git
cd loonlet-auth-assignment
```

### Step 2 — Create and Activate Virtual Environment

**On Windows:**
```bash
python -m venv venv
venv\Scripts\activate
```

**On macOS / Linux:**
```bash
python -m venv venv
source venv/bin/activate
```

### Step 3 — Install Dependencies

```bash
pip install -r requirements.txt
```

### Step 4 — Configure Environment Variables

Create a `.env` file in the root folder:

```
SECRET_KEY=your_super_secret_key_here
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30
DATABASE_URL=sqlite:///./loonlet_auth.db
```

## API Endpoints Blueprint

### Public Endpoints (No token required)

#### 1. Register User

**Route:** `POST /register`

**Request Body:**
```json
{
  "name": "Madhav",
  "email": "madhav@example.com",
  "password": "password123",
  "role": "parent"
}
```

**Success Response:**
```json
{
  "message": "User registered successfully"
}
```

---

#### 2. Login User

**Route:** `POST /login`

**Request Body:**
```json
{
  "email": "madhav@example.com",
  "password": "password123"
}
```

**Success Response:**
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5...",
  "token_type": "bearer",
  "role": "parent"
}
```

---

### Protected Endpoints (JWT token required)

Send token in every request header as:
```
Authorization: Bearer <your_access_token>
```

#### 3. Admin Portal

**Route:** `GET /admin/users`

**Access:** Admin role only

**Response:** List of all registered users

---

#### 4. Therapist Portal

**Route:** `GET /therapist/parents`

**Access:** Therapist role only

**Response:** List of all Parent-role users

---

#### 5. Parent Portal

**Route:** `GET /parent/profile`

**Access:** Parent role only

**Response:** Logged-in parent's own profile details

## Security Features

| Feature | Description |
|---|---|
| Stateless JWT Validation | Intercepts and verifies incoming authorization headers |
| Cryptographic Hashing | bcrypt-based password protection against breaches |
| Granular RBAC Guards | Dependency injection handles role filtering inline |
| Strict Input Validation | Pydantic schemas block all malformed requests |
| Secure Database Storage | Structured schema isolation for all user profiles |
| `.env` Config | Secrets never hardcoded in source code |

## Database Schema

**Table Name:** `users`

| Column | Type | Constraint |
|---|---|---|
| id | Integer | Primary Key, Auto-Increment |
| name | String(100) | Not Null |
| email | String(100) | Unique, Indexed, Not Null |
| hashed_password | String(255) | Not Null |
| role | Enum | admin / therapist / parent |
| created_at | DateTime | Auto-set on creation |

## Error Responses

| HTTP Code | Meaning |
|---|---|
| 400 | Bad Request — Email already registered or invalid input |
| 401 | Unauthorized — Token missing, invalid, or expired |
| 403 | Forbidden — Valid token but insufficient role |
| 404 | Not Found — Resource does not exist |
| 422 | Unprocessable Entity — Request body validation failed |
