##  Core Architecture & Theoretical Concepts

### 1. RESTful API Architecture
* **Statelessness:** The API does not persist client session states on the server. Every incoming request must contain all the credentials necessary to authenticate itself, ensuring the backend can scale horizontally.
* **Separation of Concerns:** Data persistence, business logic validations, and access controls are fully contained within the backend, independent of any downstream user interface.

### 2. Cryptographic Password Hashing
* **One-Way Cryptographic Salting & Hashing:** Plain-text passwords are never saved. Using `Passlib` with `bcrypt`, passwords undergo a secure, computationally expensive hashing process combined with a unique salt to mitigate rainbow table and brute-force attacks.
* **Verification Flow:** During a login event, the incoming plain-text password is systematically compared against the stored hash safely without reversing the original string.

### 3. JWT (JSON Web Token) Security Structure
A valid **JWT** consists of three distinct, dot-separated blocks:
* **Header:** Explicitly defines the hashing algorithm (e.g., `HS256`) and the token type.
* **Payload:** Holds the public claims (the user's email ID, system unique key, token expiration timestamp `exp`, and assigned role). *Note: Payload data is base64-encoded, not encrypted; sensitive items like raw passwords are never injected here.*
* **Signature:** Generated by cryptographically mixing the header, payload, and a highly secure server-side **Secret Key**. This guarantees token integrity; if a client attempts to manually forge or alter their user role, the server-side signature validation fails instantly.

### 4. Role-Based Access Control (RBAC) & PoLP
The system enforces the **Principle of Least Privilege (PoLP)**. Permissions are mapped exclusively to dedicated system roles, which are subsequently granted to users:
* **Admin Role:** Full global visibility. Can interact with endpoints to audit and view all registered system users.
* **Therapist Role:** Locked down to specialized endpoints managing clinical and therapist-specific work workflows.
* **Parent Role:** Restricts the user tightly to their personal profile metadata and dependent-level data views.

### 5. Dependency Injection Architecture
FastAPI's native `Depends` system handles access guardrails globally:
* **Token Extraction & Parsing:** Automatically intercepts the `Authorization: Bearer <token>` string from inbound HTTP request headers.
* **Inline Authorization Guards:** Before execution passes to the endpoint's core logic, sub-dependencies decode the token, check expiration bounds, pull user data, and strictly verify if the user's explicit role meets the target route's permissions. Any violation instantly triggers an automated `401 Unauthorized` or `403 Forbidden` response.

### 6. Relational Object Mapping (ORM)
* **SQLAlchemy Abstraction Layer:** Maps Python classes directly to database schemas, providing type safety, data integrity, and automatic SQL injection defense via built-in query parameterization.

Security Features
Stateless JWT Validation - Intercepts and verifies incoming authorization headers safely.

Cryptographic Hashing - Secures application records against credential breach vulnerabilities.

Granular RBAC Guards - Uses dependency injection to handle role filtering inline.

Strict Input Validation - Leverages Pydantic schemas to block malformed requests.

Secure Local Database Storage - Structured schema isolation for user profiles.

ID : Integer (Primary Key, Auto-Increment)

Email : String (Unique Constraint, Indexed)

Password : String (Securely Salted Cryptographic Hash)

Role : String (Defines RBAC mapping permissions)
